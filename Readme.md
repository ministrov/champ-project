# Манифест фронтенд-разработчика: Кодекс качества

Этот манифест — соглашение между разработчиком и пользователем. Это набор принципов, основанных на проверенных критериях, которые направляют нас в создании веб-интерфейсов, которые не просто функциональны, а исключительны. Мы не просто пишем код — мы строим цифровой рубеж с точностью, заботой и глубоким уважением к своему ремеслу.

---

## I. Визуальное соглашение: Pixel Perfect и адаптивная гармония

Мы верим, что дизайн — это обещание, а наш код — исполнение этого обещания.

- **Принцип 1.1: Точность Pixel Perfect (с пониманием).** Наша работа будет зеркально отражать замысел дизайнера. На указанных вьюпортах (мобильном, планшетном, десктопном) наша реализация будет соответствовать предоставленным макетам. Мы допускаем погрешность в **5 пикселей по горизонтали и 10 пикселей по вертикали** — снисхождение к несовершенству физического мира, но стремимся к идеальному совпадению. Для проверки мы будем использовать инструменты Pixel Perfect, Zeplin или Pixel Glass при 100% масштабе системы.
- **Принцип 1.2: Гибкое полотно.** Контентная область будет соблюдать заданные границы макета. Фоновые элементы могут растягиваться до краёв вьюпорта, но основная структура останется центрированной и ограниченной. **Мы никогда не допустим появления горизонтальной прокрутки ни на одном брейкпоинте.** Страница течёт, как вода, адаптируясь к своему контейнеру.
- **Принцип 1.3: Адаптивная целостность.** Наши решения будут по-настоящему адаптивными, корректно работая на мобильных устройствах, планшетах и десктопах, а не только при изменении ширины окна браузера. Это включает правильно настроенный метатег вьюпорта.
- **Принцип 1.4: Адаптивный логотип.** Логотип будет адаптироваться к мобильному, планшетному и десктопному состояниям проекта в соответствии с макетом.

## II. Структурная дисциплина: Семантичный и надёжный код

Наш код — это фундамент. Он должен быть чистым, семантичным и устойчивым к хаосу контента.

- **Принцип 2.1: Семантическая структура.** Мы будем использовать правильный инструмент для каждой задачи.
  - `<a>` с `href` — для навигации между страницами и якорных ссылок. `<button>` с явно указанным `type` — для действий на странице.
  - Формы будут иметь атрибут `action` и содержать все свои поля внутри тега `<form>`. Обязательные поля получат атрибут `required`.
  - **Все поля формы должны иметь уникальные значения атрибута `name`**, чтобы данные отправлялись корректно.
  - Подписи полей должны быть явно связаны с полями через оборачивающий `<label>` или атрибуты `for`/`id`.
  - Структурные ориентиры (`<header>`, `<main>`, `<nav>`, `<footer>`, `<aside>`) будут использоваться для создания чёткой, доступной для навигации структуры документа.
  - Мы будем поддерживать правильную иерархию заголовков (`<h1>` — `<h6>`) без пропусков уровней, обеспечивая логическую структуру для всех пользователей.
  - Каждый `<section>` должен иметь заголовок.
- **Принцип 2.2: Устойчивость к контенту.** Текст и контент не сломают наш макет.
  - Длинные слова будут переноситься; текст никогда не выйдет за пределы своего контейнера и не будет обрезан.
  - Динамические списки и сетки будут корректно обрабатывать любое количество элементов — от нуля до тысячи — не ломая вёрстку. Элементы будут переноситься и перестраиваться по мере необходимости.
  - Последние элементы в сетках будут выравниваться по направлению текста.
  - Повторяющиеся декоративные элементы будут появляться у новых пунктов списка.
- **Принцип 2.3: Чистая работа с текстом.** Мы никогда не будем использовать несколько тегов `<br>` для создания отступов между блоками. Одиночный `<br>` допустим для намеренных переносов строк внутри текста, но отступы должны задаваться через CSS.
- **Принцип 2.4: Чистота кодовой базы.**
  - Наша разметка будет проходить валидацию без ошибок (предупреждения допустимы).
  - Мы не будем оставлять закомментированный код без чёткого, осмысленного пояснения его назначения.
  - Неиспользуемые файлы и ассеты не имеют места в нашем проекте. Каждый файл должен быть оправдан.
  - Мы будем избегать ненужных обёрток и элементов, которые можно заменить на псевдоэлементы CSS.

## III. Визуальный язык: Единообразие и состояния

Визуальное представление должно быть последовательным, интерактивным и отзывчивым на действия пользователя.

- **Принцип 3.1: Точность типографики.** Семейство шрифта, насыщенность, начертание, размер, высота строки и цвет текста будут точно соответствовать дизайн-спецификациям.
- **Принцип 3.2: Жизненный цикл интерактивности.** Все интерактивные элементы (кнопки, ссылки, поля форм) будут иметь определённые и функциональные состояния:
  - `:hover` (визуальная обратная связь при наведении)
  - `:focus` (критически важно для навигации с клавиатуры, никогда не отключается без альтернативы)
  - `:active` (момент нажатия)
  - `:disabled` (чёткое указание неактивного состояния для элементов форм)
    Эти состояния будут следовать стайлгайду там, где он предоставлен, или реализованы продуманно в его отсутствие.
- **Принцип 3.3: Предсказуемое взаимодействие.** При взаимодействии пользователя с элементом (наведение, нажатие) ни сам элемент, ни окружающие его блоки не будут «прыгать» или смещаться, если такое поведение явно не предусмотрено макетом.

## IV. Универсальный опыт: Доступность и кроссплатформенность

Мы создаём для всех, на любых устройствах. Это не опция, а фундаментальное требование.

- **Принцип 4.1: Доступность по умолчанию.**
  - **Графика:** Все теги `<img>` будут иметь осмысленный атрибут `alt`. Декоративные изображения будут реализованы через CSS.
  - **Векторная графика:** Для контентных SVG (например, логотипа) мы добавим `role="img"` и `aria-label` с описанием.
  - **Интерактивность:** Кнопки и ссылки будут иметь текстовое описание — либо видимое, либо скрытое с помощью класса `visually-hidden`. Кнопки только с иконкой всегда будут иметь доступное имя.
  - **Фокус:** Все интерактивные элементы будут иметь видимый индикатор фокуса.
  - **Области клика:** Интерактивные элементы будут иметь щедрую и ненавязчивую область клика/касания, обеспечивая удобство использования для всех.
  - **Порядок фокуса:** Порядок перехода по фокусу будет соответствовать логической визуальной структуре страницы и не будет произвольно изменён через CSS.
- **Принцип 4.2: Кроссбраузерная гармония.** Наша работа будет одинаково выглядеть и функционировать в последних стабильных версиях **Chrome и Firefox**. Мы допускаем незначительные типографические различия, связанные с операционными системами, но гарантируем единообразие макета и базовой функциональности.
- **Принцип 4.3: Стабильность Baseline.** Мы будем преимущественно использовать CSS-свойства и JavaScript-функции, входящие в категорию **Baseline: widely available** (стабильная поддержка в основных браузерах не менее 2,5 лет). При использовании новых возможностей мы обеспечим надёжный fallback для сохранения базовой функциональности для всех пользователей.

## V. Искусство работы с ассетами: Изображения, иконки и шрифты

Мы относимся к нашим ассетам с той же заботой, что и к коду, оптимизируя производительность и качество отображения.

- **Принцип 5.1: Целевое использование графики.**
  - **Контентная графика** (информативная) размещается в HTML с помощью `<img>` и атрибута `alt`.
  - **Декоративная графика** (косметическая) размещается в CSS в качестве фоновых изображений.
  - **Исключение:** Декоративная SVG-графика, внешний вид которой нужно менять через CSS, может быть встроена в HTML.
- **Принцип 5.2: Оптимальные форматы.**
  - Используем `JPEG` для фотографий.
  - Используем `SVG` для векторных иллюстраций и иконок.
  - Используем `PNG` для всего остального (особенно для графики с прозрачностью).
  - Используем `WebP` с fallback в формате JPEG/PNG для современной производительности.
  - Декоративные векторные изображения будут объединены в спрайт.
- **Принцип 5.3: Чёткость на всех экранах.**
  - Мы подготовим графику для экранов со стандартной и удвоенной (2x) плотностью пикселей, используя `srcset` для контентных изображений и медиавыражения с `min-resolution` для фоновых.
- **Принцип 5.4: Целостность векторной графики.** Все SVG-элементы будут иметь указанные ширину и высоту в HTML или в самом теге `<svg>` (целыми числами, без единиц `px`), чтобы избежать смещения макета. Для адаптивных изображений, использующих `<picture>` с `<source>`, размеры в финальном `<img>` могут быть опущены.
- **Принцип 5.5: Типографическая независимость.**
  - Мы будем подключать шрифты локально из папки `fonts`, используя форматы `woff2` и `woff` (с указанием `woff2` первым).
  - Критически важные шрифты будут предзагружаться через `<link rel="preload">` для повышения производительности.
  - Мы будем использовать `font-display: swap` для предотвращения отображения невидимого текста во время загрузки.
  - Мы не будем полагаться на внешние сервисы, такие как Google Fonts, для основных шрифтов проекта.
  - Начертания шрифтов будут управляться через `font-weight`, а не через подключение отдельных семейств шрифтов.
  - При каждом указании шрифта будет добавляться резервное общее семейство (`serif`, `sans-serif`, `monospace`).
- **Принцип 5.6: Стандарты фавиконок.** Каждая страница будет содержать фавиконку. Файл `favicon.ico` будет размещён в корне проекта (подключать его необязательно). Фавиконка размером 32×32 в формате PNG будет подключена через `<link rel="icon" type="image/png">`.

## VI. Мастерство кода: Поддерживаемость и производительность

Наш код написан для нас и для разработчиков, которые будут работать с ним после нас. Он должен быть чистым, производительным и доставлять удовольствие от работы.

- **Принцип 6.1: Линтеры и стандарты.** В нашем проекте будут использоваться `eslint` и `stylelint` с единым конфигурационным файлом. Команда `npm run lint` будет выполняться без ошибок.
- **Принцип 6.2: Именование и организация.**
  - Все имена папок, файлов и классов будут на английском языке, в нижнем регистре, с разделением слов дефисами (`-`) или подчёркиваниями (`_`) последовательно.
  - Имена файлов и папок не будут содержать пробелов, заглавных букв или непоследовательных разделителей.
  - Мы будем использовать осмысленные, полные названия без сокращений (кроме общепринятых).
  - Цифры в названиях файлов допустимы только в конце для списков (например, `product-1.jpg`, `product-2.jpg`).
- **Принцип 6.3: Архитектура стилей.**
  - Стили будут подключаться в `<head>` одним файлом через `<link rel="stylesheet">` — никаких `@import`, никаких инлайновых стилей в `<style>` или `style=""`, за исключением демонстрации поведения JavaScript.
  - Сторонние стили (шрифты, библиотеки) будут подключаться до наших собственных стилей.
  - Глобальные селекторы (`*`, `body`, `img`) запрещены, за исключением специальных случаев: универсальный селектор только для изменения боксовой модели одновременно с псевдоэлементами, нормализация `<img>` (`max-width: 100%`), стилизация уникальных тегов `<body>`/`<html>` и использование `:root` только для кастомных свойств.
  - При использовании препроцессоров мы не будем вкладывать селекторы для создания новых имён классов (например, `.block__element` внутри `.block`) и не будем использовать функцию `@extend`/`:extend`, чтобы избежать раздувания кода и неочевидной трансформации.
- **Принцип 6.4: Производительные страницы.** Каждая страница будет набирать не менее **70 баллов в Lighthouse во всех категориях** для мобильной и десктопной версий.
- **Принцип 6.5: Модульный JavaScript.**
  - Наш JavaScript будет написан в формате ES2015-модулей, каждый в отдельном файле, названном в нижнем регистре с дефисами (например, `game-view.js`).
  - Константы (значения, известные до выполнения программы) будут именоваться в формате `UPPER_SNAKE_CASE` и группироваться в начале модуля.
  - Мы будем избегать глобальных переменных и значений — все экспорты и импорты будут явными.
  - Мы будем избегать дублирования кода (принцип DRY) — повторяющаяся логика будет вынесена в функции.
  - Поиск элементов по селекторам будет выполняться минимальное количество раз, с сохранением ссылок на элементы для повторного использования (не внутри циклов).
  - Для каждого события будет использоваться отдельный обработчик (одна функция не будет обрабатывать разные типы событий).
  - Длинные функции и методы будут разбиты на несколько небольших.
  - Оператор присваивания не будет использоваться как часть выражения.
- **Принцип 6.6: Управление памятью и событиями.**
  - Обработчики событий, добавленные к глобальным объектам (`document`, `window`), будут своевременно добавляться и удаляться для предотвращения утечек памяти.
  - Обработчики не будут многократно добавляться при скрытии и показе элементов — они будут добавлены один раз и управляться флагами, либо удаляться когда не нужны.
  - В процессе выполнения кода не должно возникать необработанных ошибок.
- **Принцип 6.7: Кроссбраузерный JavaScript.** Наш код будет работать без ошибок в различных браузерах и операционных системах.

## VII. Обязательство перед качеством

Этот манифест — наш ориентир, наш стандарт и наше обязательство. Мы следуем этим принципам не как простому чек-листу, а как отражению нашей профессиональной гордости. Мы создаём для веба с намерением, гарантируя, что каждый пользователь, на любом устройстве, получит опыт, который будет красивым, функциональным и доступным.
